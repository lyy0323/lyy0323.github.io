# /tech 内容接入指南

本文档面向 AI agent。你的任务是：浏览一个项目目录，和作者对话，共同产出一个 `.mdx` 文件。

`/tech` 下有三种内容类型：

| 类型 | 目录 | URL | 说明 |
|------|------|-----|------|
| **project** | `src/content/projects/{slug}/index.mdx` | `/tech/projects/{slug}/` | 文件夹式，可包含子页面 |
| **blog** | `src/content/blogs/{slug}.mdx` | `/tech/blog/{slug}/` | 单页面技术文章/笔记 |
| **app** | `src/content/apps/{slug}.mdx` | `/tech/app/{slug}/` | 交互式 H5/demo |

**核心理念：这不是简历，是简历的补充。** 简历上写「社员诗歌网站 PM，110w 访问」，这里要讲的是：为什么做这件事、过程中踩了什么坑、做了哪些取舍、学到了什么。读者看完应该觉得「这个人想得很清楚」，而不是「这个人很厉害」。

---

## 1. Frontmatter 字段

Project 专有字段标注 `[project]`，其余三种类型共享。

```yaml
---
title: string          # 名称。中文优先，可用「中文名 · 英文名」格式
description: string    # 一句话简介，40-80字。说人话，不要写成 README 第一段
category: enum         # [project] website | app | game | data | art | agent | tool | other
tags: string[]         # 技术栈/领域标签，3-6 个
cover: string?         # 封面图路径（可选）
url: url?              # 线上地址（可选）
repo: url?             # 代码仓库地址（可选）
status: enum           # active | archived | wip
date: string           # 启动时间，'YYYY-MM'
featured: boolean      # 重点内容，默认 false
order: number          # 排序权重，现有项目占用 1-6
---
```

## 2. 正文结构和调性

### 2.1 整体调性

- **像在和一个对你感兴趣的人聊天**，而不是在做汇报。
- 可以说「我当时其实不确定这样做对不对」「事后来看我会换一种方式」。
- 可以聊非技术的东西：动机是什么情感驱动的、哪个瞬间让你觉得这件事有意义。
- 不用每句话都在证明自己，而是在展示自己的思考方式。

### 2.2 推荐章节

按项目实际情况取舍，不要硬凑。每节可以是一段话，也可以是 bullet list，看哪个更自然。

```markdown
## 这个项目是什么
一段话，说清楚项目做的事。写给完全不了解背景的人看。

## 为什么做这个
不是「市场调研发现需求」那种话。而是：
是什么契机、什么感受、什么不满足让你开始做这件事？
这部分最能展示你作为 PM / 创作者的直觉。

## 我做了什么
具体的工作内容。可以讲技术方案，但更重要的是讲决策——
为什么选 A 不选 B？做了哪些取舍？有没有推翻过自己的方案？

## 过程中的思考
这是最有价值的部分。可以聊：
- 遇到的最大挑战是什么，怎么解决的
- 哪些假设被验证了，哪些被推翻了
- 如果重新来过，会怎么做不同
- 这个项目改变了你对某件事的看法吗

## 一些数字（可选）
如果有的话，自然地带出来。不要为了放数字而放数字。
「上线第一周有 200 个人用了，说实话比我预期的多」比
「DAU 峰值 200」有意思得多。

## Q&A

<details>
<summary>Q: [问题]</summary>

[回答]
</details>

<details>
<summary>Q: [问题]</summary>

[回答]
</details>
```

### 2.3 Q&A 部分

每个项目附 2-5 个 Q&A。这是整篇文章的点睛之笔。

**问题来源优先级：**

1. **面试官会问的问题**——但要回答得像聊天，不像面试
   - 「为什么选这个技术栈而不是 XXX？」
   - 「这个项目最难的部分是什么？」
   - 「如果用户量增长 10 倍，架构上需要改什么？」
   - 「你怎么衡量这个产品的成功？」
   - 「和团队/用户有过什么分歧，怎么处理的？」
2. **展示思维深度的问题**
   - 「做这个项目之前和之后，你对 XX 的理解有什么变化？」
   - 「如果给你无限资源，这个项目你会做成什么样？」
   - 「这个项目有没有什么'副产品'——不在计划内但意外有价值的东西？」
3. **有趣的、个人化的问题**
   - 「做这个项目期间最开心的一个瞬间是什么？」
   - 「有没有什么彩蛋或者只有你自己知道的细节？」

**回答原则：**
- 不要泛泛而谈。好的回答有具体的场景、具体的例子。
- 可以展示犹豫和不确定：「说实话我不确定当时的决定是最优的，但……」
- 长度灵活。有些问题一两句话就够，有些值得写一段。

## 3. 工作流程

### 3.1 第一步：扫描项目目录

浏览项目目录，从以下来源提取事实性信息：

```
README.md                    → 项目简介、功能
package.json / pyproject.toml → 技术栈
src/ 目录结构                 → 功能模块
git log                      → 时间线、活跃度
.github/workflows/           → 部署方式
```

### 3.2 第二步：向作者提问

**你必须和作者对话。** 代码能告诉你做了什么，但不能告诉你为什么。

扫描完目录后，向作者提出 3-6 个问题。问题应该覆盖：

- **动机层**（1-2 个）：为什么做？什么契机？
- **决策层**（1-2 个）：为什么这么做？有过什么取舍？
- **反思层**（1-2 个）：学到了什么？会改什么？

提问格式示例：

```
我浏览了 lingshu 的项目目录，看到这是一个基于 RAG 的 Agent 系统，
用了 LangChain + FastAPI，数据源包括飞书文档和诗歌数据库。

想和你聊几个问题，帮我把这个项目写得更有意思：

1. 你说灵枢是给新任社干用的——具体是什么场景让你觉得「这个东西非做不可」？
   有没有一个具体的瞬间或者事件触发了这个想法？

2. RAG + Agent 的方案是一开始就定的，还是试过别的路线？
   选 LangChain 而不是自己写 pipeline 有什么考虑？

3. 上线之后，社干实际使用的方式和你设想的一样吗？
   有没有什么出乎意料的用法？

4. 如果现在重新做这个项目，你会改什么？
```

### 3.3 第三步：写作

拿到作者的回答后，将事实信息和个人洞察编织成正文。

**关键原则：**
- 作者的原话比你的概括有力。如果作者说了一句很生动的话，直接用。
- 不要把作者的回答直接贴上去，要编辑成流畅的叙述。
- Q&A 部分可以直接基于对话中最有趣的问答来写，也可以另外构思新问题。
- 如果作者某个回答特别好但不适合放在正文里，放到 Q&A。

### 3.4 第四步：作者确认

写完后交给作者审阅。作者可能会：
- 修正事实错误
- 要求调整某些措辞（「这个说法太高调了」「这段可以更具体」）
- 补充新的内容

## 4. 完整示例

```mdx
---
title: 灵枢 · 智能知识中枢与决策辅助 Agent
description: 给国学社做的 AI 知识助手。把散落在飞书、微信群、诗歌库里的社团知识整合起来，让新社干不用再翻三年前的聊天记录。
category: agent
tags: ['AI Agent', 'LLM', 'RAG', 'Full-stack', 'Python']
status: active
date: '2025-06'
featured: true
order: 2
---

## 这个项目是什么

灵枢是一个面向上海交大国学社的 AI Agent 系统。简单来说，它把社团这些年散落在各处的知识——飞书文档、微信群聊记录、诗歌数据库、活动策划案——都整合进了一个可以对话的系统里。新上任的社干可以直接问它「去年诗歌擂台赛是怎么组织的」「社费审批流程是什么」，而不用去翻三年前的群聊。

## 为什么做这个

每年换届的时候，新社干面对的最大问题不是没有能力，而是没有上下文。国学社运转了这么多年，积累了大量的隐性知识——哪个活动的坑在哪里、和哪些社团合作过、哪种宣传方式转化率高——这些东西全在前任的脑子里。

我自己当社长的时候就体会过这种「交接失真」的痛苦。写了很长的交接文档，但接任者真正遇到问题的时候，文档里往往找不到答案——因为你没办法预判所有问题。我就想，能不能让社团的知识像一个有经验的「老社员」一样，随时可以问。

## 我做了什么

技术上是一个比较标准的 RAG + Agent 架构：用向量检索找到相关的知识片段，喂给 LLM 生成回答。但实际做起来，最花时间的不是写代码，而是数据治理——飞书文档的格式五花八门，微信群聊记录的信噪比极低，诗歌数据库的 schema 和知识库完全不同。光是把这些数据源清洗、切分、统一索引，就花了整个项目 60% 的时间。

架构上选了 LangChain + FastAPI。LangChain 在 Agent 工作流编排上省了不少事，虽然它的抽象有时候让 debug 变得很痛苦，但对于这个量级的项目来说，快速跑通比架构完美更重要。

## 过程中的思考

做灵枢最大的收获不是技术上的，而是对「知识管理」这件事有了新的理解。之前我以为知识管理的核心是「存储」——把信息放在一个地方，需要的时候去找。但实际上核心是「检索意图的理解」。用户问「去年擂台赛怎么组织的」，他真正想知道的可能是预算怎么申请、场地怎么借、评委怎么邀请——这些信息分散在五六份不同的文档里。

这个认知后来也影响了我对 AI 产品的理解：很多时候，用户说出来的需求和真正的需求之间有一层翻译。

## Q&A

> **Q: 为什么选 LangChain 而不是自己搭 pipeline？**
>
> 说实话一部分原因是时间紧。灵枢从立项到上线只有一个月，我一个人全栈开发，没有余裕去造轮子。LangChain 的 Agent 和 Tool 抽象刚好能用，虽然它的 callback 机制和版本更新经常让我抓狂。如果现在重新做，可能会考虑用更轻量的方案，比如直接用 Claude 的 tool use，省掉中间那层抽象。

> **Q: 上线之后社干的使用方式和你预期的一样吗？**
>
> 不太一样。我原本设想的是新社干用它来查历史信息，但实际上用得最多的场景是「帮我写一段活动策划案的初稿」。大家发现与其从零写，不如让灵枢根据过往的策划案生成一个骨架，然后在上面改。这算是一个意外的产品发现——用户自己找到了更高效的使用方式。

> **Q: 你怎么衡量灵枢成功不成功？**
>
> 老实说我没有设 KPI。对我来说，如果换届之后新社干还在用它，而不是打开一次之后就再也不碰了，它就是成功的。比量化指标更重要的是，它有没有真的融入社团的工作流程。目前来看还行——至少没人问我「灵枢怎么登录」了，说明大家记住了入口。
```

## 5. 从项目目录提取信息

### 5.1 快速扫描

```
README.md                             → 项目简介、功能
package.json / pyproject.toml         → 技术栈、依赖
src/ 目录结构                          → 功能模块
git log --oneline -20                 → 最近动态
git log --reverse --format='%ai' | head -1  → 启动时间
.github/workflows/                    → 部署方式
```

### 5.2 判断 status

```
最近 3 个月有 commit   → active
超过 1 年无 commit     → archived
README 写了 WIP/TODO   → wip
```

### 5.3 提取 tags

优先具体框架名（`FastAPI`）而非语言名（`Python`）。从 dependencies / imports / 文件后缀推断。3-6 个。

## 6. 输出检查清单

- [ ] frontmatter 所有必填字段都有值
- [ ] category 是枚举值之一
- [ ] tags 是 `['A', 'B']` 格式，3-6 个
- [ ] description 40-80 字，说人话
- [ ] date 格式 `'YYYY-MM'`
- [ ] order 没有和现有项目冲突（现有占用 1-6）
- [ ] 正文读起来像聊天，不像简历或 README
- [ ] 有 Q&A 部分，2-5 个问题
- [ ] Q&A 的回答有具体细节，不是泛泛而谈
- [ ] **已经和作者对话过**，不是纯粹从代码推断的内容
- [ ] 文件名：小写英文 + 连字符，如 `my-project.mdx`
